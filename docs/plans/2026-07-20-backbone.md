# Life Topography Backbone Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Establish a runnable, tested ingestion-kernel backbone that safely accepts idempotent evidence batches and advances source cursors only with committed evidence.

**Architecture:** A framework-independent connector SDK feeds application services through typed batches. The first persistence adapter is in-memory for contract tests, followed by SQLite behind an explicit unit-of-work boundary. HTTP, CLI, and MCP are thin adapters over the same services.

**Tech Stack:** Python 3.12, uv, Pydantic 2, FastAPI, SQLAlchemy 2, Alembic, Typer, official MCP Python SDK, pytest, Ruff, mypy, Docker.

---

### Task 1: Establish the workspace and doctrine

**Files:** Create root project metadata, package manifests, `AGENTS.md`, README, license, security/contribution docs, ADRs, CI, and Docker skeleton.

**Verification:** `uv lock && uv sync --all-packages --dev`; metadata commands complete without dependency or packaging errors.

### Task 2: Define the connector contract using TDD

**Files:**
- Test: `packages/sdk/tests/test_contracts.py`
- Create: `packages/sdk/src/life_topography_sdk/models.py`
- Create: `packages/sdk/src/life_topography_sdk/connector.py`

**RED:** Assert capability validation, stable evidence identity, opaque cursor round-trip, and rejection of empty batches.

**GREEN:** Implement the smallest Pydantic models and async connector Protocol satisfying those behaviors.

**Verification:** focused SDK test passes, followed by full suite.

### Task 3: Define transactional ingestion behavior using TDD

**Files:**
- Test: `packages/app/tests/test_ingestion_kernel.py`
- Create: `packages/app/src/life_topography/application/ingestion.py`
- Create: `packages/app/src/life_topography/ports/evidence_store.py`
- Create: `packages/app/src/life_topography/adapters/memory_store.py`

**RED:** Prove duplicate batches are idempotent and a failed evidence write does not advance the cursor.

**GREEN:** Implement a small unit-of-work port, ingestion receipt, and in-memory adapter.

**Verification:** focused kernel tests and full suite pass.

### Task 4: Add SQLite persistence and migrations using TDD

**Files:** storage integration tests, SQLAlchemy tables/adapter, Alembic configuration and initial migration.

**RED:** Run import/reopen/idempotency/cursor-atomicity tests against a temporary database.

**GREEN:** Implement WAL, foreign keys, busy timeout, unique source identities, and one transactional writer.

**Verification:** migration from empty DB and repeated upgrade both succeed; persistence tests pass.

### Task 5: Add thin operational surfaces

**Files:** FastAPI health/source-status endpoints, Typer `doctor` and fixture-import commands, read-only MCP source/status/search tools.

**RED:** Adapter tests assert they call application services and do not expose credentials, arbitrary SQL, or filesystem paths.

**GREEN:** Wire the same service container to all three surfaces.

**Verification:** API test client, CLI runner, and MCP protocol tests pass.

### Task 6: Add a synthetic reference connector

**Files:** fixture connector, synthetic records, SDK contract tests, import-to-query end-to-end test.

**Verification:** bootstrap, interruption/resume, duplicate delivery, incremental change, reconciliation, and deletion fixtures pass without network or real personal data.

### Task 7: Make deployment boring

**Files:** multi-stage/non-root Dockerfile, Compose file, `.env.example`, health check, persistent `/data` volume, CI workflow, release notes template.

**Verification:** build image; start Compose; import fixture; query status; restart; confirm persistence. If Docker is unavailable, CI syntax and local commands may be verified, but Docker runtime validation remains explicitly blocked.

### Task 8: Scaffold sanity pass and publish

Run formatting, lint, types, tests, package builds, documented examples, secret scan, and Git diff review. Publish only intended files to a new repository owned by the user. Record the exact commands and real results.
